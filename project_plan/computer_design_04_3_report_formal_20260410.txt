第1章 作品概述
1.1 创意来源与项目背景
本作品的创意来源于团队对海南橙类品质分级、产地识别和品牌表达场景的持续关注。在前期调研中，团队发现果园、合作社、分选中心和品牌方虽然都对“分级更稳定、产地更可信、结果可回查”有明确需求，但现实流程里仍然存在明显断点：一方面，品质判断往往依赖人工经验，不同人员、不同场地之间容易出现口径偏差；另一方面，即便已经形成检测结果，也常常停留在纸面记录、聊天沟通或单次结论层面，难以沉淀为可复查、可留档、可持续调用的数字材料。
基于这一问题，团队提出“橙源智鉴”这一作品方向，尝试把样本文件、分析流程、报告输出和历史档案组织为一套可演示、可联调、可继续扩展的系统。作品并不把目标停留在单一模型展示或单页界面效果上，而是希望围绕橙类样本的上传、分析、结果说明和后续复盘，构建一条相对完整的数字化链路，使分析结果真正能够进入分级、验货和品牌说明等后续动作。
1.2 面向用户与应用场景
本作品主要面向四类用户。第一类是果园和合作社，这类用户更关注采样登记、批次记录和分级依据是否统一；第二类是分选中心和初加工主体，这类用户更关注分析速度、异常提醒和出货前复核效率；第三类是品牌运营方和渠道采购方，这类用户更关注产地说明、结果留档和标准化报告能否直接用于沟通与验货；第四类是区域服务机构与科研团队，这类用户更看重样本积累、档案沉淀和后续扩展能力。
对应到应用场景，本作品适合首先进入海南橙类的试点场景，例如澄迈福橙与琼中绿橙相关的采样、批次分析、品质说明和样本归档工作。在这一阶段，作品强调的是“能展示、能验证、能继续扩样”；随着样本规模和验证深度的进一步提升，系统还可以继续向区域品牌服务、科研辅助和更广泛的农产品质控场景扩展。
1.3 作品构成与阶段成果
当前作品由网站端、小程序端、业务后端和模型服务四部分组成。网站端承担项目展示、分析结果可视化、模型说明和历史档案查看；小程序端承担采样登记、文件上传、分析结果查看、报告回看等移动端任务；业务后端承担认证、样本管理、文件管理、分析任务调度、报告导出等业务流程；模型服务承担预测接口与联调演示能力。四部分既可独立运行，也可在当前工程中形成完整链路。
截至2026年4月10日，网站端与小程序端均已形成可操作界面，后端与模型服务可完成联调，核心页面与接口具备继续演示和扩展的基础。根目录网站已完成生产构建验证，小程序后端已完成 TypeScript 构建验证，Python 模型服务样例已完成编译检查。与前期单纯的方案说明相比，当前作品已经具备较完整的工程形态。

1.4 应用价值与阶段判断
从当前阶段看，本作品的价值主要体现在三个方面。第一，它把原本分散的样本文件、分析结论和报告材料组织成连续流程，使结果更容易进入实际经营动作；第二，它同时支持网站端与小程序端，能够兼顾项目展示与现场轻量操作；第三，它已经具备后端、模型服务、文件管理和PDF导出的工程基础，说明作品不只是概念展示，而是具备继续迭代和试点的现实条件。

第2章 问题分析
2.1 问题来源
橙类产品在进入市场流通前，通常都会经历采样、分选、验货、包装、品牌说明和售后复核等多个环节。问题并不在于行业没有数据，而在于这些数据往往没有被放进统一的数字流程中。人工经验分级虽然灵活，但缺少稳定可复用的判断口径；理化或实验室检测虽然更适合作为验证依据，但周期较长、批量能力有限；品牌方和渠道方在获得结果后，也常常缺少结构化的批次档案和可直接对外使用的说明材料。
在品牌果、电商果和礼盒果场景中，上述问题会被进一步放大。用户不只关心“这颗果子怎么样”，更关心“这一批货为什么这样判断、依据是什么、后续还能不能回查”。如果系统无法同时支撑分析、说明与留档，那么即使分析结果本身存在，也很难真正进入经营链路。
2.2 现有解决方案分析
当前相关解决方案大致可以分为三类。第一类是人工经验分级，这类方案部署成本较低、操作门槛较低，但判断结果高度依赖个体经验，跨场景和跨人员复用能力较弱；第二类是理化或实验室检测，这类方案在验证与仲裁层面更有依据，但难以承接现场高频、批量、快速判断任务；第三类是面向外观缺陷或尺寸分选的通用视觉系统，这类方案在外观识别上较有优势，但对内部品质、产地差异和批次留档的支撑能力仍然有限。

2.3 本作品要解决的核心痛点
- 分级判断缺乏统一口径，同一批次结果受人员差异影响明显。
- 产地说明多停留在纸面材料或口头表达，缺乏可回看、可对照的数字证据。
- 分析结果难以继续进入报告、档案、验货和复盘等后续流程，信息利用率不高。
- 网站展示、移动端采样、业务管理和模型服务之间缺少统一承接，难以形成完整闭环。
2.4 解决问题的总体思路
本作品采用“样本文件接入—批次关联—模型分析—报告生成—历史归档”的总体思路。首先，在数据接入层支持 HDR、RAW、CSV 等文件类型，并围绕批次信息组织样本与文件关系；其次，在业务层组织等级识别、产地判断和品质指标估计等多任务分析；再次，在结果层将模型输出继续整理为可展示的结构化结果与文本化建议；最后，在应用层通过网站端和小程序端把结果送入历史档案、报告导出和现场回看流程。
这一思路的核心不在于单独强调某一种算法，而在于让数据、页面、接口和报告四条链路能够相互对应。也正因如此，报告第3章和第4章会分别从技术方案与工程实现两个角度继续展开，而第5章将用当前项目中的构建、联调和页面展示结果来说明这一思路已经具备可验证基础。
第3章 技术方案
3.1 总体技术路线
作品整体采用分层式技术架构。前端分为网站端和微信小程序端，分别服务展示分析与移动操作；中间层为 Node.js 业务后端，负责会话管理、样本管理、文件处理、任务调度和报告导出；模型层为独立 Python HTTP 服务，负责健康检查与预测接口输出；底层通过 Prisma 与数据库组织用户、产地、样本、光谱文件、分析任务、分析结果和报告文件等数据关系。
技术路线可概括为：样本采集与理化整理形成基础数据，光谱文件进入系统后完成样本关联和元数据管理，业务后端根据任务类型发起同步或异步分析，模型服务返回预测结果，再由后端整合为分析结果、结构化报告与 PDF 导出内容，最终在网站端和小程序端进行展示、查询和回看。
3.2 数据基础与样本组织
当前项目已整理海南两地代表性样本数据，重点覆盖澄迈福橙和琼中绿橙两类对象。根据项目现有数据整理模块，当前已沉淀 399 条理化样本记录，其中澄迈福橙 199 条、琼中绿橙 200 条；已整理 8 类典型数据文件，涵盖化验值、液质指标以及 HSI、R210、S960 等不同类型光谱文件；同时已整理 6 组代表性样本用于页面演示和对比说明。这些数据为作品当前阶段的展示、联调和报告编写提供了相对明确的事实基础。

在样本指标上，两地样本已形成一定差异性特征。例如，现有数据整理结果显示，琼中绿橙平均糖度和平均糖酸比高于澄迈福橙，而澄迈福橙在维生素 C 平均值上表现更高。这些差异为产地识别、品质判断和对比展示提供了较好的切入点，也使作品能够在页面和报告中呈现更有层次的分析内容。
3.3 多端协同与系统架构设计
网站端采用 Next.js 16 与 React 19 构建，目前已形成 9 个主要页面，包括首页、关于页、分析页、数据大屏页、评级页、历史页、模型页、产地页和政策页。网站端更偏向项目展示、分析结果可视化、数据说明和档案回看，是作品对外展示和管理分析的主要入口。
小程序端采用原生微信小程序实现，目前已形成 11 个页面，包括 home、upload、analysis、analyzing、result、report、history、samples、sample-detail、insights、profile 等模块。与网站端相比，小程序端更强调移动场景下的轻量操作，适合现场采样、结果查询和报告回看。
业务后端采用 Express、Prisma 与 Zod 组织接口和数据校验。后端已覆盖登录、首页、洞察、用户中心、产地、样本、文件、模型配置、同步分析、异步任务、结果详情、报告读取与 PDF 导出等核心能力。根据当前项目文档，核心接口数量已达到 20 个，能够支撑前后端并行联调与后续持续扩展。
模型服务采用 Python 与 FastAPI 实现独立 HTTP 服务样例，当前提供 `/health` 与 `/predict` 两个接口，并支持通过 `MODEL_PROVIDER=PYTHON_HTTP` 与 Node 后端切换联调。这一设计使模型层与业务层保持解耦，便于后续将样例服务替换为更真实的推理能力。
3.4 关键技术与增量部分
与单一页面演示型作品相比，本作品的技术增量主要体现在四个方面。第一，形成网站端、小程序端、后端和模型服务的完整多端协同架构，而不是只展示某一端页面；第二，引入异步分析任务状态流转机制，使任务能够经历 `PENDING → RUNNING → SUCCESS/FAILED` 的完整过程，更贴近真实业务系统；第三，加入文件上传、鉴权下载和批次关联机制，使光谱文件不再只是离线资料，而是可进入业务流程的对象；第四，加入结构化报告与 PDF 导出能力，使结果不仅停留在接口返回，而能进一步转换为可查看、可传递的正式材料。
需要特别说明的是，当前模型服务仍以样例推理和联调验证为主，但这种“业务架构先完整、模型能力持续替换”的做法，能够保证作品在当前阶段就具备完整工程闭环，而不是等待最终模型完全成熟后才开始组织系统实现。
第4章 系统实现
4.1 网站端实现
网站端已围绕项目展示和分析可视化形成相对完整的页面体系。首页用于呈现作品主题、入口和核心卖点；分析页用于演示单样本分析和对比分析流程，展示上传、进度、结果卡片与智能分析报告；数据大屏页用于展示样本规模、准确率、产地比较和异常预警等可视化内容；历史页用于查看样本记录、筛选和导出；模型页、产地页和政策页则进一步补充模型说明、产地背景与政策支撑信息。
其中，分析页已经形成较完整的演示闭环：页面支持单样本与对比分析模式切换，能够展示分析进度、光谱曲线与智能报告内容；数据大屏页则聚焦统计总览、样本特征空间分布和实时检测动态；历史页则围绕样本编号、预测产地、糖度、酸度、糖酸比和品质评级组织表格化展示。尽管当前部分页面仍使用整理后的代表性数据或演示数据驱动，但页面逻辑和结果表达方式已经基本成形。
4.2 小程序端实现
小程序端的实现重点在于“现场轻量使用”。当前已形成 11 个页面，覆盖首页、上传、分析、分析中、结果、报告、历史、样本详情、样本列表、洞察和个人中心等模块。根据当前后端联调进度，小程序分析流程已支持查看当前样本已上传文件数量，并在未上传光谱文件时禁止直接发起分析；任务提交后会进入独立的“分析中”动画页，而不是停留在原页面轮询；结果页与报告页已能够展示更结构化的分析结论。
从工程组织上看，小程序项目已与网站端、后端和模型服务拆分成独立工作区，便于并行开发与后续部署。相比将所有能力都压在一个 Web 页面上的方案，这种拆分更适合参赛作品在演示与试点之间保持灵活切换。
4.3 业务后端实现
业务后端在当前作品中承担中枢作用。首先，在数据层，Prisma schema 已对用户、产地、样本、光谱文件、分析任务、分析结果和报告文件等实体进行了建模；其次，在接口层，系统已覆盖微信登录、首页数据、洞察、样本详情、文件上传、分析任务创建、异步任务轮询、结果详情、报告读取、PDF 导出和模型配置等能力；再次，在安全层，系统已接入基础鉴权与用户归属校验，保证用户中心、分析、报告和上传等接口在登录后按归属访问。
在任务执行机制上，后端已形成同步快捷分析与异步任务两套路径。`POST /api/analysis/scan` 适合详情页快速触发单样本分析，而 `POST /api/analysis/tasks` 则用于异步受理正式任务，并通过任务状态、进度、错误信息、结果ID和报告ID组织完整任务链路。这一机制不仅提高了工程完整性，也为后续接入更复杂的模型服务预留了空间。
在文件与报告层，后端已支持本地文件上传、文件元数据落库、带鉴权的文件下载地址下发，以及按既有报告内容动态生成 PDF。PDF 生成模块能够将任务信息、预测结果、综合评级、产地结论、指标解读和建议组织为中文可读报告，使分析输出具备更正式的交付形式。

4.4 模型服务实现
模型服务当前采用 FastAPI 样例实现，提供健康检查和预测接口。服务内部定义了任务类型、样本状态、产地信息和指标信息等结构，并基于请求内容返回预测产地、置信度、糖度、酸度、糖酸比、维生素 C 与 AI 摘要等字段。这一设计的意义不在于宣称当前模型已经完全成熟，而在于先把模型服务作为可替换的独立组件接入系统，使网站端和小程序端能够通过统一接口完成联调和展示。
在业务后端联动下，系统已能够通过 `GET /api/model/config` 获取当前模型配置，并通过 `POST /api/model/predict/preview` 或分析任务接口完成预览与任务调用。模型服务当前版本为 `python-http-demo-1.0.0`，并支持通过环境变量切换鉴权和接口地址。这些内容已经足够支撑参赛作品在“模型服务已形成独立模块”这一层面的工程表达。
4.5 部署与运行实现

根据当前后端阶段进度文档，系统已完成登录、首页、洞察、用户中心、产地、样本、文件、模型配置、同步分析、异步任务、结果详情、报告读取与 PDF 导出等核心链路联调。现有文档记录显示，文件上传链路、模型预览接口、任务详情读取、报告 PDF 响应头与下载功能都已完成阶段性验证；未登录访问用户中心与文件下载会返回 `401`，跨用户访问任务、结果和报告会返回 `403`，说明基础安全控制已经进入可用状态。
从接口组织上看，当前系统已形成 20 个核心 API 接口，足以支撑网站端与小程序端的主要交互需求。相比只展示静态页面的作品，这说明本作品已经从“有页面”发展到了“有数据、有接口、有状态流转”的阶段。
5.4 数据与结果展示分析
在数据支撑方面，当前项目已沉淀 399 条理化样本，覆盖澄迈福橙与琼中绿橙两类对象，并整理了 8 类典型数据文件。前端展示中，系统可展示海南两地平均糖度、平均酸度、平均糖酸比与平均维生素 C 的差异信息，也可基于代表性样本呈现单样本分析与对比分析结果。这些内容使作品在“样本基础”上不再停留于抽象设想，而具备可具体说明的数据支撑。
在结果表达方面，当前页面展示口径显示，单次分析流程约为 1.2 秒，产地识别置信度可达 98.5%，SSC 预测误差展示为 ±0.3%。这些结果主要用于说明当前原型系统已经具备较完整的展示与联调能力。报告在表述时将其明确界定为“当前页面演示口径”，避免把阶段性结果直接写成最终性能结论。
5.5 测试结论与当前边界
综合本章测试可以得出三点结论。第一，作品当前已经具备完整工程链路，网站端、小程序端、后端和模型服务可以分别运行并形成整体配合；第二，作品的结果输出不再停留于单个数值，而能够进一步进入结果页、报告页和 PDF 导出流程；第三，当前作品已经具备继续补强样本、模型和试点场景的现实基础。
同时也需要看到作品当前的边界。其一，模型服务目前仍以样例推理和联调验证为主，后续仍需替换为更完整的真实模型；其二，现有样本与演示数据虽然足以支撑原型系统说明，但要形成更强说服力，仍需要补充更大规模样本、跨批次对比和误差分析；其三，网站端部分可视化页面仍带有演示属性，后续需要继续提升真实业务数据接入比例。正因为存在这些边界，正式版报告在整体表述上会有意避免过度承诺。
第6章 作品总结
6.1 作品特色与创新点
- 围绕真实业务链路组织作品，而不是只停留在单页展示或单次模型输出。
- 形成网站端、小程序端、后端和模型服务四层协同的完整工程结构。
- 支持异步分析任务、文件上传下载、结构化结果展示和 PDF 报告导出。
- 在项目表述上保留阶段边界，强调“可验证、可试点、可扩展”，避免把演示结果写成最终定论。
6.2 应用推广
从应用推广角度看，本作品适合首先进入果园、合作社、分选中心和品牌方等对批次分析、结果说明和档案留存有明确需求的场景。当前作品已经具备网站展示、小程序操作、业务后端和报告导出的完整基础，因此可以作为区域试点和场景验证的起点使用。随着样本规模、模型能力和实际合作案例继续增加，系统还可以进一步向区域品牌服务机构、科研辅助和更多农产品质控场景扩展。
6.3 作品展望
后续工作重点将放在四个方面。第一，继续补强区域样本库、理化标签和典型文件，使模型和报告拥有更扎实的数据基础；第二，进一步完善跨批次、跨场景的稳定性验证，补充更多对比实验和误差分析；第三，继续提高网站端和小程序端中真实业务数据的接入比例，减少演示属性较强的展示内容；第四，系统化整理参赛所需的源代码、部署文件、模型、数据说明、网盘链接与版权信息，使作品在提交材料层面更加完整规范。

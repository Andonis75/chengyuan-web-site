第1章 作品概述
1.1 项目背景与创意来源
海南柑橘产业在品质分级、产地识别和批次管理等环节长期存在“检测结果分散、判断标准不统一、档案复用效率低”的现实问题。围绕这一痛点，本作品聚焦海南澄迈福橙与琼中绿橙两类代表性样本，构建集数据治理、品质检测、产地溯源、结果解读与报告输出于一体的智能分析平台，力求把果品检测从单次结论提升为可追踪、可复盘、可服务业务决策的数据链路。
作品以真实样本、统一实验流程和完整工程系统为基础，形成面向果园、合作社、分选中心、品牌运营方的数字化检测方案。平台以多模态光谱数据为主要输入，结合滴定法与液质数据构建标签体系，在同一系统中完成上传、建模、分析、展示、归档和报告生成，体现出较强的工程完成度和应用转化价值。
1.2 面向对象与应用场景
本作品面向四类典型用户：一是果园和合作社，用于样本登记、产地批次管理和品质快速筛查；二是分选加工与质检环节，用于对糖度、内部成分和产地信息进行辅助判断；三是品牌运营与渠道验货环节，用于形成标准化报告和批次凭证；四是科研与区域服务机构，用于沉淀样本、验证模型、扩展更多柑橘检测任务。
在应用场景上，平台既支持实验环境下的系统化研究，也支持面向移动端的现场操作。平台输出不止停留在页面展示层，而是能够进一步生成结构化分析结果、历史记录和 PDF 报告，为品控、宣传、验货和复核提供统一依据。
1.3 系统组成与建设成果
当前作品已形成“展示与分析前端、移动采样终端、业务后端、模型服务”四层协同架构。项目主站基于 Next.js 16 与 React 19 构建，已形成首页、分析页、数据大屏、历史页、模型页等 9 个主要页面；微信小程序已形成 11 个功能页面，覆盖上传、分析、结果、报告、样本与个人中心等操作入口；后端完成用户、样本、文件、分析任务、结果和报告等核心业务承接；模型服务完成健康检查、预测接口和联调能力封装。
截至 2026 年 4 月 10 日，网站端生产构建、后端 TypeScript 构建与 Python 模型服务编译检查均已通过，项目已具备稳定演示、联调验证和持续扩展的基础。围绕样本管理、分析任务、结果展示与报告输出，作品已经形成较为完整的系统闭环。

1.4 作品价值与总体定位
本作品的价值体现在三个方面。第一，平台把分散在实验记录、光谱文件和化验结果中的信息重新组织为可复用的数据资产，提升检测结果的管理效率和可解释性。第二，平台将多模态样本分析与工程系统实现结合起来，不仅能给出判断结果，还能完成结果展示、报告生成与档案留存。第三，平台以海南特色柑橘为起点，具备向更多柑橘品种、更多品质指标和更多区域检测任务扩展的潜力。
第2章 数据基础与问题分析
2.1 业务问题分析
传统果品质量判断高度依赖人工经验与离线检测结果，往往存在批次信息分散、检测周期偏长、结果复用率低等问题。特别是在品牌果、电商果和礼盒果等场景中，用户不仅关注“结果是什么”，更关注“依据是什么”“是否能回查”“能否形成统一说明”。因此，单点检测工具难以满足当前对品质判断、产地识别与数据留存同步推进的需求。
本作品将问题拆分为三个可落地任务：其一，建立可用于品质预测与产地溯源的标准化样本主表；其二，形成面向业务使用的多模态分析流程；其三，将实验结果转化为面向实际场景的系统化输出，使分析结果能够真正进入验货、归档和品牌说明流程。
2.2 数据基础与样本组织
根据当前正式实验资料，项目已完成 399 个样本的正式主表整理，覆盖澄迈福橙与琼中绿橙两个产地。正式实验统一采用 R210 与 S960 双光谱作为建模输入，其中 R210 光谱特征 228 维，S960 光谱特征 1402 维，合计 1630 维。围绕平台任务，项目构建了品质预测、产地溯源、液质成分预测三类任务数据集，并统一使用固定随机种子形成训练、验证、测试三组划分，确保实验结果具备可复现性与可比性。
在样本组成方面，澄迈福橙样本 199 个，琼中绿橙样本 200 个；统一划分下训练集 279 个、验证集 60 个、测试集 60 个，且两地产地在验证集与测试集各保持 30:30 的均衡分布。这一组织方式使平台既能支撑产地分类任务，也能为后续品质与成分建模提供稳定的数据基础。

2.3 数据治理与质量控制
本作品在数据治理上坚持“先做数据质量控制，再做模型训练”的原则。项目清洗结果显示，HSI 数据存在跨产地同编号完全重复问题，共识别出 199 对异常样本对。为保证正式实验的可靠性，项目在正式建模主表中完全剔除了 HSI，仅保留质量稳定、可统一对齐的 R210 与 S960 双光谱数据，形成“安全版_去除HSI融合表”，这是后续所有正式实验与系统展示共用的数据基座。
在多模态对齐方面，项目完成了 S960 两地产地数据的共同波段统一，其中共同波段数为 1402 个，琼中原始数据额外尾部波段 50 个，通过共同波段对齐后保证了不同产地样本可直接进入统一建模流程。该过程体现了平台在数据工程层面的严谨性，也为后续结果的可解释性提供了保障。

2.4 核心突破点
围绕上述问题，本作品重点突破四个方面：一是建立面向真实业务的标准化样本体系；二是形成覆盖品质预测、产地溯源和液质成分预测的多任务分析路线；三是将数据治理、模型推理与工程系统实现打通；四是把分析结果组织为可展示、可下载、可归档的正式输出。也就是说，本作品的突破点不在于单个模型分数，而在于从样本到报告的全链路建设。
第3章 技术方案
3.1 总体架构设计
平台整体采用分层架构。前端层负责交互展示与操作承载，服务于项目展示、样本分析和结果回看；业务层负责鉴权、样本管理、文件管理、任务调度和报告导出；模型层负责统一接收分析请求并返回预测结果；数据层通过 Prisma 管理用户、产地、样本、光谱文件、分析任务、分析结果和报告文件等核心实体，保证系统逻辑、数据状态与输出材料保持一致。
这一架构设计的优势在于将建模能力和业务能力解耦。模型可以持续更新，业务系统和展示系统不需要频繁重构；反过来，前后端和报告链路也不依赖某一个固定算法版本，从而保证作品在比赛展示、后续迭代和场景落地中都具有较好的稳定性。
3.2 多模态数据与建模路线
在数据层，平台以双光谱安全主表为核心，融合滴定法品质指标与液质成分标签，形成“样本编号 - 产地信息 - 品质标签 - 光谱特征 - 历史记录”的统一数据对象。在建模层，平台同步推进三类任务：任务1为品质预测，围绕糖度、酸度、糖酸比和 VC 建立品质评价框架；任务2为产地溯源，面向澄迈福橙与琼中绿橙建立二分类识别模型；任务3为液质成分预测，围绕 36 项液质成分建立内部组分建模能力，并从中筛选强响应指标作为重点分析对象。
在实验组织方式上，项目采用统一样本划分和多模型对比策略，首先建立双光谱与单光谱基线，再引入 SNV 预处理、PCA 降维、岭回归、PLS 回归、SVM、XGBoost 等方法进行优化。这样既保证了实验过程的完整性，也使最终报告能够清晰呈现从数据治理、特征处理到任务建模的完整技术链条。
3.3 关键算法与优化策略
项目实验表明，针对高维光谱数据，预处理与特征压缩是提升表现的关键。平台在液质成分预测任务中重点采用 SNV 标准正态变量变换削弱散射与基线漂移影响，再结合 Ridge、PLS 与 PCA 完成回归建模。进阶优化结果显示，液质成分预测任务的验证集平均 R² 已提升至 0.7104，测试集平均 R² 稳定在 0.68 以上，说明双光谱融合对内部成分表征具有较强潜力。
在品质预测任务中，平台围绕糖度、VC、糖酸比等指标开展定向建模。测试结果显示，糖度指标的最佳测试集 R² 达到 0.7051，表现出较好的实际参考价值；VC 与糖酸比分别达到 0.3778 和 0.3654，可作为辅助判断依据。通过“核心指标重点优化、综合指标统一评价”的方式，平台形成了既符合科学性又适合工程展示的品质分析表达。
表3-1 三项核心任务与技术路线
任务 | 输入 | 输出 | 技术路线
品质预测 | R210/S960 双光谱 | 糖度、酸度、糖酸比、VC | SNV + Ridge/PLS 等回归方法
产地溯源 | 单光谱或双光谱 | 澄迈/琼中产地判别 | 逻辑回归、线性 SVM、LightGBM 分类
液质成分预测 | 双光谱安全主表 | 36 项液质成分与重点指标 | SNV + PCA/Ridge/PLS/XGBoost

3.4 业务流程与输出设计
平台围绕“样本接入 - 文件上传 - 模型分析 - 结果展示 - 报告生成 - 历史归档”构建完整业务链路。用户完成样本登记后，可上传相应文件并发起分析请求，系统通过同步分析与异步任务两种方式承接不同负载场景。结果返回后，平台一方面提供面向页面的可视化展示，另一方面自动组织为结构化结论和 PDF 报告，满足比赛展示与业务交付的双重需求。
第4章 系统实现
4.1 展示端与移动端实现
平台的交互层采用“网站端 + 微信小程序”双入口设计。网站端重点承担作品总览、数据大屏、分析展示、模型说明和历史记录等任务，适合用于评审展示、成果汇报和后台查看；微信小程序重点承载样本上传、分析提交、结果查看和报告回看等轻量化操作，适合在现场采样或移动使用场景中直接调用。两端在业务逻辑与数据结构上保持一致，由统一后端提供支撑，避免了重复开发与信息割裂。
在页面组织上，平台围绕“看得见、讲得清、能操作”三项原则构建界面逻辑。分析页面提供单样本分析与对比分析入口，数据大屏集中展示样本规模、产地差异和关键指标变化，历史页面负责管理分析记录与报告输出，从而使作品展示、功能操作与结果留存三者保持连贯。
4.2 后端与模型服务实现
业务后端采用 Express、Prisma 与 Zod 组织，实现了用户登录、样本管理、文件上传、模型配置、同步分析、异步任务、结果详情、报告读取和 PDF 导出等核心能力。根据当前项目资料，系统已形成 20 个核心 API，可同时支撑网站端与小程序端的主要交互流程。数据库层围绕用户、样本、文件、任务、结果和报告进行统一建模，保证页面状态、任务状态和输出结果的一致性。
模型服务采用 Python FastAPI 独立实现，当前提供健康检查与预测接口，并支持通过配置项与 Node 后端联调。这样的拆分方式使模型层能够持续迭代，而业务系统仍保持稳定接口，对参赛作品后续升级和场景部署都具有积极意义。
4.3 任务调度与报告输出实现
系统在分析执行机制上同时支持快速预览与异步任务两条路线。对需要即时反馈的场景，可通过同步接口快速返回分析结果；对需要完整记录和后续追踪的场景，系统通过任务状态流转机制完成排队、执行、成功或失败等状态管理，并将分析结果与报告文件绑定归档。该设计既贴近真实业务系统的运行方式，也提升了作品在比赛演示时的完整度与可信度。
在结果输出层，系统可将样本信息、产地判断、品质指标、核心成分解释和综合结论组织为中文报告，并生成正式 PDF 文件。这样一来，作品输出不再停留于“模型给出一个值”，而是形成可阅读、可传递、可留档的正式材料，增强了平台的交付能力。

第5章 实验与测试分析
5.1 实验设置
本作品的实验部分以正式双光谱主表为基础，在统一划分条件下开展对比测试。训练集、验证集、测试集规模分别为 279、60、60；其中两类产地样本在验证集与测试集均保持均衡分布，保证分类与回归结果都具有较好的横向可比性。实验采用先基线、后优化的组织方式，在保障结果真实可信的前提下逐步提升任务表现。
5.2 产地溯源结果分析
在产地溯源任务中，平台取得了最稳定、最亮眼的结果。以 S960单光谱 为输入时，逻辑回归与线性 SVM 在验证集与测试集上的准确率、平衡准确率与 F1_macro 均达到 1.0000，说明当前双产地样本在光谱空间中具有较高可分性。这一结果为平台在产地鉴别、品牌溯源和批次核验方面提供了坚实支撑。

5.3 品质预测结果分析
在品质预测任务中，糖度是当前最具代表性的优势指标。测试结果显示，R210+S960双光谱 条件下的 岭回归 模型在糖度预测上取得测试集 R²=0.7051，RMSE=0.8071，能够为果品分级和品质判断提供较强参考。与此同时，VC 和糖酸比任务也取得了一定效果，适合作为综合分析中的辅助信息，帮助平台形成更完整的品质画像。
从两地产地的平均理化差异看，琼中绿橙在糖度和糖酸比上更高，分别达到 10.79 和 18.11；澄迈福橙在 VC 水平上更高，平均达到 40.80。这些真实样本差异不仅增强了平台的分析解释力，也为页面展示与报告撰写提供了事实支撑。

5.4 液质成分预测结果分析
液质成分预测是本作品最能体现“多模态数据价值”的任务之一。基线实验中，双光谱融合岭回归模型已取得验证集平均 R²=0.6811、测试集平均 R²=0.6620 的结果。进一步引入 SNV 预处理与参数搜索后，液质成分任务的最佳验证集平均 R² 提升至 0.7104，最佳测试集平均 R² 达到 0.6975，说明双光谱路线在内部成分建模上具备持续优化空间。
在重点成分上，异柠檬酸、莽草酸等指标表现突出：异柠檬酸测试集最佳 R² 达到 0.8225，莽草酸达到 0.8024，蔗糖达到 0.6773。这表明平台不仅能完成外部品质判断，还具备向内部组分感知延伸的能力，从而提升了作品在“检测 + 解释”层面的综合竞争力。

第6章 特色创新与推广前景
6.1 作品特色与创新点
本作品的创新首先体现在“数据治理先行”的研究范式上。面对多模态样本中存在的异常模态与重复问题，团队将异常识别、模态筛选、统一划分和安全主表重建前置到正式实验之前，夯实了后续建模与系统展示的基础。这种从源头提升数据可信度的做法，是作品能够形成说服力的重要原因。
第二个创新点在于将多模态光谱分析与工程系统深度结合。作品不仅完成了双光谱建模和多任务实验，还同步实现了样本管理、异步任务、结果展示、历史归档和 PDF 报告输出，形成了从样本到报告的完整闭环。第三个创新点在于“检测结果可解释化”，平台能够把产地、品质、成分等多类信息组织为结构化输出，更适合面向实际应用场景落地。

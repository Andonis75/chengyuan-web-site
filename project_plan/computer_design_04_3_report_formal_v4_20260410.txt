第1章 作品概述
1.1 项目背景与创意来源
在海南柑橘的分级、验货和品牌传播环节，常见的问题不是“没有数据”，而是数据分散、判断口径不统一、结果难以长期留存和复用。围绕这一现实需求，项目选取澄迈福橙与琼中绿橙作为代表样本，搭建起集数据治理、品质检测、产地溯源、结果解读和报告输出于一体的分析系统，希望把一次检测结果真正沉淀为可追踪、可复盘、可支撑业务决策的数据资产。
项目以真实样本、统一实验流程和完整工程实现为基础，面向果园、合作社、分选中心和品牌运营等场景提供数字化检测方案。系统以多模态光谱为主要输入，结合滴定法和液质数据建立标签体系，在同一套流程中完成上传、建模、分析、展示、归档和报告生成，既强调检测能力，也强调实际可用性。
1.2 面向对象与应用场景
系统主要面向四类用户：一是果园和合作社，用于样本登记、批次管理和品质初筛；二是分选加工与质检环节，用于辅助判断糖度、内部成分和产地信息；三是品牌运营与渠道验货环节，用于生成标准化报告和批次凭证；四是科研与区域服务机构，用于沉淀样本、验证模型并拓展更多柑橘检测任务。
从使用场景看，系统既能支撑实验环境下的系统化研究，也能服务现场采样和移动端操作。输出结果不仅体现在页面上，还能进一步沉淀为结构化记录、历史档案和 PDF 报告，为品控、宣传、验货和复核提供统一依据。
1.3 系统组成与建设成果
当前系统由主站、小程序、业务后端和模型服务四部分组成。主站基于 Next.js 16 与 React 19 开发，已完成首页、分析页、数据大屏、历史页、模型页等 9 个主要页面；微信小程序已完成 11 个功能页面，覆盖上传、分析、结果、报告、样本和个人中心等入口；后端负责用户、样本、文件、分析任务、结果和报告等核心业务；模型服务负责健康检查、预测接口和联调支撑。
截至 2026 年 4 月 10 日，主站生产构建、后端 TypeScript 构建与 Python 模型服务编译检查均已通过。围绕样本管理、分析任务、结果展示和报告输出，系统已经打通了完整流程，具备稳定演示和持续完善的基础。

1.4 作品价值与总体定位
这项工作的价值，首先在于把分散在实验记录、光谱文件和化验结果中的信息重新整理成可复用的数据资产，让检测结果更容易管理、也更容易解释；其次，它把样本分析和工程实现真正接到了一起，输出的不再只是一个判断值，还包括展示、归档和报告生成；再往后看，以海南特色柑橘为起点搭起来的这套方法，也为扩展到更多品种、更多指标和更多区域任务留出了空间。
第2章 数据基础与问题分析
2.1 业务问题分析
传统果品质量判断仍然较多依赖人工经验和离线检测。这样做虽然能得到结论，但常常伴随批次信息分散、检测周期偏长、结果难复用等问题。尤其在品牌果、电商果和礼盒果场景中，用户关心的不只是“结果是什么”，还包括“依据是什么”“能不能回查”“能不能统一对外说明”。单一检测工具很难同时满足这些需求。
围绕这些问题，项目把任务拆成三部分推进：先建立可用于品质预测和产地溯源的标准化样本主表；再搭建面向业务使用的多模态分析流程；最后把实验结果整理成适合验货、归档和品牌说明的系统化输出。
2.2 数据基础与样本组织
根据当前正式实验资料，项目已完成 399 个样本的正式主表整理，覆盖澄迈福橙与琼中绿橙两个产地。正式实验统一采用 R210 与 S960 双光谱作为输入，其中 R210 光谱特征 228 维，S960 光谱特征 1402 维，合计 1630 维。围绕三项核心任务，项目进一步整理出品质预测、产地溯源、液质成分预测三类数据集，并使用固定随机种子完成训练、验证、测试划分，保证实验过程可复现、结果可对比。
在样本组成方面，澄迈福橙样本 199 个，琼中绿橙样本 200 个；统一划分后训练集 279 个、验证集 60 个、测试集 60 个，且两地产地在验证集和测试集中都保持 30:30 的均衡分布。这样的组织方式既适合做产地分类，也能为后续品质和成分建模提供稳定基础。

2.3 数据治理与质量控制
项目在数据治理上坚持先清洗、后训练。清洗结果显示，HSI 数据存在跨产地同编号完全重复问题，共识别出 199 对异常样本。为保证正式实验可靠，项目在正式主表中完全剔除了 HSI，仅保留质量稳定、可以统一对齐的 R210 与 S960 双光谱数据，形成“安全版_去除HSI融合表”，作为后续实验和系统展示共用的数据底座。
在多模态对齐方面，项目完成了 S960 两地产地数据的共同波段统一，其中共同波段数为 1402 个，琼中原始数据额外尾部波段 50 个。完成共同波段对齐后，不同产地样本可以直接进入统一建模流程，也为后续结果解释打下了基础。

2.4 核心突破点
围绕上述问题，项目并没有把重点放在单一功能上，而是把样本体系、多任务建模、数据治理、工程实现和结果输出一起往前推。换句话说，这个作品的优势不只在某一项分数上，而在于从样本进入系统，到结果生成报告，整条链路已经能够闭合运行。
第3章 技术方案
3.1 总体架构设计
系统整体采用分层架构，但并不是为了追求结构好看，而是为了把展示、业务、模型和数据各自放在最合适的位置。前端负责交互与展示，业务层负责样本、文件、任务和报告流转，模型层统一接收分析请求并返回预测结果，数据层则通过 Prisma 管理用户、产地、样本、光谱文件、分析任务、分析结果和报告文件等核心实体，让系统状态和输出材料始终对得上。
这样的架构把建模能力和业务能力拆分开来。模型可以持续更新，业务系统和展示系统不需要频繁重构；前后端与报告链路也不依赖某一个固定算法版本，便于后续迭代和场景落地。
3.2 多模态数据与建模路线
在数据层，系统以双光谱安全主表为核心，融合滴定法品质指标与液质成分标签，形成“样本编号 - 产地信息 - 品质标签 - 光谱特征 - 历史记录”的统一数据对象。在建模层，项目同步推进三类任务：任务1为品质预测，围绕糖度、酸度、糖酸比和 VC 建立评价框架；任务2为产地溯源，面向澄迈福橙与琼中绿橙建立二分类识别模型；任务3为液质成分预测，围绕 36 项液质成分探索内部组分建模能力，并筛选强响应指标作为重点分析对象。
在实验组织上，项目采用统一样本划分和多模型对比策略，先建立双光谱与单光谱基线，再引入 SNV 预处理、PCA 降维、岭回归、PLS 回归、SVM、XGBoost 等方法继续优化。这样既保证了实验过程完整，也使报告能够清晰展示从数据治理、特征处理到任务建模的技术链条。
3.3 关键算法与优化策略
实验结果表明，对于高维光谱数据，预处理和特征压缩是影响效果的关键环节。液质成分预测任务中，项目重点采用 SNV 标准正态变量变换削弱散射与基线漂移，再结合 Ridge、PLS 与 PCA 进行回归建模。进阶优化后，液质成分预测任务的验证集平均 R² 提升到 0.7104，测试集平均 R² 稳定在 0.68 以上，说明双光谱融合对内部成分具有较好的表征能力。
在品质预测任务中，项目围绕糖度、VC、糖酸比等指标开展定向建模。测试结果显示，糖度指标的最佳测试集 R² 达到 0.7051，是当前表现最稳定、最适合直接服务分级判断的指标；VC 与糖酸比分别达到 0.3778 和 0.3654，可作为综合分析中的辅助依据。这样的结果结构更符合实际使用场景，也更便于页面和报告表达。
表3-1 三项核心任务与技术路线
任务 | 输入 | 输出 | 技术路线
品质预测 | R210/S960 双光谱 | 糖度、酸度、糖酸比、VC | SNV + Ridge/PLS 等回归方法
产地溯源 | 单光谱或双光谱 | 澄迈/琼中产地判别 | 逻辑回归、线性 SVM、LightGBM 分类
液质成分预测 | 双光谱安全主表 | 36 项液质成分与重点指标 | SNV + PCA/Ridge/PLS/XGBoost

3.4 业务流程与输出设计
系统围绕“样本接入 - 文件上传 - 模型分析 - 结果展示 - 报告生成 - 历史归档”组织业务流程。用户完成样本登记后，可上传文件并发起分析请求；系统通过同步分析与异步任务两种方式承接不同场景。结果返回后，一部分用于页面展示，另一部分自动整理为结构化结论和 PDF 报告，便于展示、交付和留档。
第4章 系统实现
4.1 展示端与移动端实现
在交互入口上，项目没有把所有操作都压在一个界面里，而是拆成主站和微信小程序两部分。主站更适合展示作品全貌、查看数据大屏、回看历史结果；小程序则更贴近日常使用，适合现场上传样本、提交分析、查看结果和回看报告。两端共用同一套业务逻辑和数据结构，因此使用场景不同，但信息不会断开。
在页面组织上，系统围绕“看得见、讲得清、能操作”三项原则展开。分析页面提供单样本分析与对比分析入口，数据大屏集中展示样本规模、产地差异和关键指标变化，历史页面负责管理分析记录与报告输出，使展示、操作和留存三部分保持连贯。
4.2 后端与模型服务实现
业务后端采用 Express、Prisma 与 Zod 组织，实现了用户登录、样本管理、文件上传、模型配置、同步分析、异步任务、结果详情、报告读取和 PDF 导出等核心能力。根据当前项目资料，系统已形成 20 个核心 API，可以同时支撑主站和小程序的主要交互流程。数据库层围绕用户、样本、文件、任务、结果和报告进行统一建模，保证页面状态、任务状态和输出结果保持一致。
模型服务采用 Python FastAPI 独立实现，当前提供健康检查与预测接口，并支持通过配置项与 Node 后端联调。这样的拆分方式使模型层可以持续迭代，而业务侧接口保持稳定，便于后续升级与部署。
4.3 任务调度与报告输出实现
系统在分析执行机制上同时支持快速预览和异步任务两条路线。对需要即时反馈的场景，可通过同步接口快速返回结果；对需要完整记录和后续追踪的场景，系统通过任务状态流转机制完成排队、执行、成功或失败等管理，并将分析结果与报告文件绑定归档。这样的设计更贴近真实业务运行方式，也增强了比赛展示时的完整度。
在结果输出层，系统可将样本信息、产地判断、品质指标、核心成分解释和综合结论组织成中文报告，并生成正式 PDF 文件。结果不再停留在“模型给出一个值”，而是能够转化为可阅读、可传递、可留档的正式材料。
4.4 运行与验证情况
项目当前已完成主站 `npm run build`、后端 `npm run build` 与 Python 模型服务 `python -m compileall` 验证，说明系统不只有展示效果，也具备基本工程交付能力。各模块职责清晰，能够支撑比赛演示、后续开发和扩展部署。

第5章 实验与测试分析
5.1 实验设置
实验部分没有采用临时抽样或随意拆分的方式，而是全部基于正式双光谱主表，在统一划分条件下完成对比测试。训练集、验证集、测试集规模分别为 279、60、60；两类产地样本在验证集和测试集中保持均衡分布，这样做的目的很直接，就是让分类和回归结果都能放在同一把尺子下比较。整个实验按“先基线、后优化”的顺序推进，结论也更容易复核。
5.2 产地溯源结果分析
产地溯源是当前表现最稳的一项任务。以 S960单光谱 为输入时，逻辑回归与线性 SVM 在验证集和测试集上的准确率、平衡准确率与 F1_macro 均达到 1.0000，说明两类产地样本在当前光谱空间中的区分度已经比较清晰，这也让产地鉴别、品牌溯源和批次核验有了很直接的结果支撑。

5.3 品质预测结果分析
在品质预测任务中，糖度是当前最有代表性的优势指标。测试结果显示，R210+S960双光谱 条件下的 岭回归 在糖度预测上取得测试集 R²=0.7051，RMSE=0.8071，可以为果品分级和品质判断提供较强参考。与此同时，VC 和糖酸比也取得了一定效果，适合作为综合分析中的辅助信息，帮助系统形成更完整的品质画像。
从两地产地的平均理化差异看，琼中绿橙在糖度和糖酸比上更高，分别达到 10.79 和 18.11；澄迈福橙在 VC 水平上更高，平均达到 40.80。这些差异让页面展示和报告结论都有了更扎实的事实依据。

5.4 液质成分预测结果分析
液质成分预测最能体现多模态数据的价值。基线实验中，双光谱融合岭回归模型已取得验证集平均 R²=0.6811、测试集平均 R²=0.6620。进一步引入 SNV 预处理和参数搜索后，液质成分任务的最佳验证集平均 R² 提升至 0.7104，最佳测试集平均 R² 达到 0.6975，说明双光谱路线在内部成分建模上还有明确的提升空间。
在重点成分上，异柠檬酸、莽草酸等指标表现尤其突出：异柠檬酸测试集最佳 R² 达到 0.8225，莽草酸达到 0.8024，蔗糖达到 0.6773。换句话说，系统已经不只是停留在外部品质判断上，而是开始能对内部组分给出更细的识别结果。

5.5 综合测试结论
综合实验和工程验证结果可以看出，项目已经搭起了“数据可治理、模型可验证、系统可运行、结果可交付”的完整框架。产地溯源任务表现突出，品质预测在糖度等核心指标上已经能服务实际判断，液质成分预测也把分析深度继续往里推了一步。更重要的是，这些结果都建立在统一数据主表、统一样本划分和完整工程系统之上，所以它不是零散结果的拼接，而是一套能够前后对上的作品。
第6章 特色创新与推广前景
6.1 作品特色与创新点
这个项目最值得强调的一点，是没有一上来就追模型结果，而是先把数据基础做扎实。面对多模态样本中的异常模态与重复问题，团队先完成异常识别、模态筛选、统一划分和安全主表重建，再进入正式实验。这样做虽然更花时间，但后面的建模和展示都更稳，也更能经得住追问。
另一点创新，在于没有把算法和系统实现分开做。项目在推进双光谱建模和多任务实验的同时，也同步完成了样本管理、异步任务、结果展示、历史归档和 PDF 报告输出，把从样本到报告的整条流程接了起来。再往前走一步，检测结果也不只是给出数值，而是能把产地、品质、成分等信息整理成结构化内容，实际使用时会顺手很多。
表6-1 作品创新点概述
创新点 | 具体体现
数据治理创新 | 识别 HSI 异常并重建安全主表，确保正式实验可信
多任务融合创新 | 同步支持品质预测、产地溯源、液质成分预测
工程闭环创新 | 实现上传、分析、展示、归档、报告导出全流程
结果表达创新 | 将模型结果转化为可阅读、可传递、可留档的正式报告

6.2 应用推广前景
就当前完成度来看，项目最适合先落在海南特色果品的品质检测和品牌溯源场景。系统已经具备样本管理、任务分析、结果回看和报告输出等基础能力，可以支撑区域试点、产地展示、品牌说明和课程教学演示。后续只要样本规模继续扩大、更多品类接入，推广到更多柑橘品种乃至其他农产品场景也是顺理成章的。
6.3 总结
回到这份作品本身，橙源智鉴想解决的不是一个孤立的识别问题，而是把海南柑橘从样本整理、分析判断到结果输出这条链路真正做顺。现在这版成果已经有了扎实的数据基础、较明确的实验结果，也完成了比较完整的系统实现和展示。拿来参加大数据实践赛，它既有可讲清楚的技术内容，也有可以现场展示的系统形态，整体上是站得住的。
